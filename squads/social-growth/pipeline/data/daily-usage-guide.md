# Daily Usage Guide

## For the Team

### Chat Terminal

- When working in this chat as the terminal, use `chat-operating-pattern.md` and `client-suggestion-registry.md` to keep the response format aligned with the current database records.

1. Open the current client and product records in the database before planning anything.
2. Check whether any offer, audience or tone updates were added.
3. Run research only after the persisted records are consistent.
4. Persist the research result in the database before planning strategy.
5. Create strategy only after consulting the persisted strategy library.
6. Run the GEO / AI discoverability audit when the site or entity is still ambiguous.
7. Treat audience and persona as hypotheses derived from evidence, not fictional personas.
8. Do not run trend research during market research or strategy; start it only in the content phase (`Step 03AA` blog topic backlog).
9. Build the blog architecture when the cycle includes long-form content, including the word-count target, featured-image direction and free/public source search note.
10. Create the blog draft from the approved architecture, keeping the approved length, image constraints and source rationale intact.
11. Optimize discovery content for SEO, GEO and LLM readability before repurposing.
12. Repurpose the approved blog into native social assets when multichannel output is needed.
13. Create content from the approved strategy.
14. Review performance and update the client record when something changes.
15. Use `daily-execution-checklist.md` to keep execution and monitoring consistent, including source metadata for external imagery.
16. Use `account-launch-kit.md` and `weekly-cycle-template.md` when starting a new client or a new cycle.
17. Use `internal-quickstart.md` when the team needs a one-page operating reference.
18. Use `operational-index.md` when you need the fastest route to any artifact.
19. Use `start-here.md` when you want the shortest possible entry point.
20. Use `master-guide.md` when you want the whole flow in one short read.
21. Use `naming-standard.md` before creating new reusable docs or templates.
22. Use `research-and-proposal-route.md` as the canonical structure for any market or social research output.
23. Use `social-sales-proposal-module.md` only when the client confirms the proposal checkpoint or explicitly asks for a proposal; the first client-facing output must be the presentation deck with title and objective.
24. When the delivery needs to be client-friendly, place the final result in a client-named folder as a single consolidated `client-report.md`.
25. Use `social-intelligence-intake.md` when the system needs to discover and score social profiles automatically.
26. Use `social-intelligence-summary-template.md` when you need the structured output of the social audit before proposal building.

## For the Client

- send new links or documents when the offer changes;
- correct any wrong extracted information directly in the client record;
- keep priority services updated;
- add new proof points and cases when they become available.
- confirm when research should open a new market version and when it should reuse the current one.

## Rule of Thumb

If the client record changes, the strategy may need a refresh.
If the strategy changes, the content and schedule should follow.
If the research is only analytic and no proposal was requested, stop before commercial packaging.
If execution gets repetitive, review the daily checklist before changing the process.
