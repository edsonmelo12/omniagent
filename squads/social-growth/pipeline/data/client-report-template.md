# Client Report Template

## Purpose

Create a single consolidated client report that lives inside a client-named folder and combines:
- social intelligence;
- market research;
- proposal readiness and, when the proposal checkpoint is confirmed, the commercial proposal/deck;
- evidence and source notes.

## Output Location

Use the pattern:
- `output/<client-slug>/client-report.md`

Example:
- `output/dallas-rent-a-car/client-report.md`

## Required Sections

1. **Executive Summary**
   - what the client is
   - what the market looks like
   - what the main gap is
   - whether the material is presentation-ready

2. **Client Profile**
   - name
   - website
   - segment
   - main offer
   - audience
   - tone of voice
   - context or geography

3. **Social Intelligence**
   - profiles discovered
   - public signals
   - confidence notes
   - data gaps and fallback notes

4. **Market Research**
   - market frame
   - competitive landscape
   - strategic reading

5. **Score Readout**
   - presence
   - consistency
   - proof
   - conversion
   - maturity
   - confidence

6. **Opportunities**
   - 3 to 5 actionable openings

7. **Proposal Readiness**
   - whether the evidence supports a presentation
   - whether it supports a proposal
   - what is missing if not ready
   - whether this cycle opted into the consolidated commercial report

8. **Risks and Assumptions**
   - uncertainty
   - validation needs
   - operational assumptions

9. **Recommendation**
   - thesis
   - archetype
   - presentation objective
   - recommended direction
   - KPIs

10. **Proposal Deck**
   - slide 1: cover + objective
   - slide 2: current state
   - slide 3: diagnosis
   - slide 4: market
   - slide 5: transformation
   - slide 6: recommended offer
   - slide 7: next steps

11. **Evidence**
   - source URLs
   - observed dates
   - source notes
   - proposal checkpoint result, when applicable

## Rules

- one client, one consolidated report file;
- the report should not be split across separate research/proposal/deck documents for the final delivery;
- keep confirmed metrics and inferred metrics labeled;
- do not invent numbers;
- if the proposal checkpoint is confirmed, the presentation header must appear before the long-form narrative;
- if the proposal checkpoint is not confirmed, omit the deck section and keep the report focused on research, strategy and evidence;
- use the same client slug across the folder and the report filename when possible.

## Example Structure

```md
output/<client-slug>/client-report.md
```

The consolidated report may still reference supporting artifacts, but the client-facing deliverable should live in a single document.
